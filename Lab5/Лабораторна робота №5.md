- **Перелік функціональних залежностей:** визначити та перелічити функціональні залежності (ФЗ) для кожної проблемної таблиці.
- **Перевірка нормальних форм:** оцінити поточну нормальну форму у кожної таблиці (1NF, 2NF, 3NF) на основі її функціональних залежностей (ФЗ) та структури ключа.
- **Застосування нормалізації:** перетворення таблиць у вищі нормальні форми (до ЗНФ) для усунення часткових та транзистивних залежностей.

## Поточні таблиці
---
- `users` - користувачі
- `developers` - розробники
- `employers` - роботодавці
- `projects` - проєкти

## SQL скрипт бази даних
---
```sql
DROP TABLE IF EXISTS projects;
DROP TABLE IF EXISTS developers;
DROP TABLE IF EXISTS employers;
DROP TABLE IF EXISTS users;

CREATE TABLE users (
	user_id SERIAL PRIMARY KEY,
	user_name VARCHAR(30) NOT NULL,
	user_surname VARCHAR(50) NOT NULL,
	user_country VARCHAR(100) NOT NULL,
	user_email VARCHAR(100) NOT NULL UNIQUE,
	user_phone VARCHAR(18) NOT NULL
);

CREATE TABLE developers (
	dev_id INT PRIMARY KEY REFERENCES users(user_id) ON DELETE CASCADE,
	is_searching_job BOOLEAN NOT NULL DEFAULT TRUE,
	dev_links TEXT
);

CREATE TABLE employers (
	employer_id INT PRIMARY KEY REFERENCES users(user_id) ON DELETE CASCADE,
	employer_company VARCHAR(150) NOT NULL,
	employer_position VARCHAR(100) NOT NULL
);

CREATE TABLE projects (
	project_id SERIAL PRIMARY KEY,
	dev_id INT NOT NULL REFERENCES developers(dev_id) ON DELETE CASCADE,
	project_name VARCHAR(150) NOT NULL,
	project_description TEXT NOT NULL,
	project_url VARCHAR(255) NOT NULL,
	project_category VARCHAR(50) NOT NULL,
	project_active BOOLEAN NOT NULL DEFAULT TRUE
);
```

## Аналіз початкової схеми та функціональні залежності
---
Спочатку проаналізуємо поточну схему бази даних та приклади даних у них

**Функціональні залежності для початкової схеми**
 - `users` : `user_id` -> `user_name`,`user_surname`, `user_country`, `user_email`, `user_phone`
 - `developers`: `dev_id` -> `is_searching_job`, `dev_links`
 - `employers` : `employer_id` -> `employer_company`, `employer_position`
 - `projects`: `project_id` -> `dev_id`, `project_name`, `project_description`, `project_url`, `project_category`, `project_active`

**Поточна нормальна форма (до нормалізації)** 
На даний момент, моя початкова схема не знаходиться навіть у 1NF. Головна проблема - таблиця `developers`. Поле `dev_links` має тип `TEXT` і призначене для зберігання кількох посилань через кому (наприклад github та linkedin одночасно). Це порушує правило атомарності атрибутів. Окрім цього, у таблицях `employers` та `projects` є текстові поля, які будуть дублюватися і можуть викликати аномалії оновлення.

## Нормалізація до 1NF
---
Всі атрибути повинні містити лише атомарні значення. Масиви чи списки не допускаються.

Потрібно видалити `dev_links` з таблиці `developers` і створити нову підпорядковану таблицю `developer_links` щоб кожне посилання розробника зберігалося в окремому рядку.
```sql
CREATE TABLE developer_links (
	link_id SERIAL PRIMARY KEY,
	dev_id INT NOT NULL REFERENCES developers(dev_id) ON DELETE CASCADE,
	link_url VARCHAR(255) NOT NULL
);

ALTER TABLE developers
DROP COLUMN dev_links;
```

## Нормалізація до 2NF
---
Для нормалізації до 2NF таблиця повинна бути в 1NF і не повинна міститит часткових залежностей. На даний момент часткові залежності можуть існувати лише в таблицях зі складеними первинними ключами. Так як у моїй базі даних усі таблиці мають одинарні первинні ключі `user_id`, `dev_id`, `employer_id`, `project_id`, то часткових залежностей тут не може бути. Це значить, що після нормалізації до 1NF моя база даних технічно перейшла одразу до 2NF.

## Нормалізація до 3NF
---
Для переходу до 3NF таблиця повинна бути у 2NF і не містити транзитивних залежностей. Також на цьому етапі повинне бути усунене дублювання (надлишковість даних).

1.У таблиці `employers` є поле `employer_company` (назва компанії). Багато роботодавців (Hr-ів) можуть бути з одної компанії. Якщо назва компанії змінеться, доведеться оновлювати велику кількість рядків (аномалія оновлення). Я виношу компанію в окрему таблицю `companies`. 2. Також у таблиці `projects` є поле `project_category`. Щоб уникнути помилок при введені назв категорій та дублювання тексту, потрібно створити окрему таблицю `categories` і додати зовнішній ключ у проєкти.

```sql
CREATE TABLE companies (
	company_id SERIAL PRIMARY KEY,
	company_name VARCHAR(150) NOT NULL UNIQUE
);

ALTER TABLE employers
ADD COLUMN company_id INT REFERENCES companies(company_id) ON DELETE SET NULL;

ALTER TABLE employers
ADD COLUMN employer_company;

```

```sql
CREATE TABLE categories (
	category_id SERIAL PRIMARY KEY,
	category_name VARCHAR(50) NOT NULL UNIQUE
);

ALTER TABLE projects
ADD COLUMN category_id INT REFERENCES categories(category_id) ON DELETE SET NULL;

ALTER TABLE projects
DROP COLUMN project_category;
```

## SQL скрипт бази даних після нормалізації
---
```sql
DROP TABLE IF EXISTS projects;
DROP TABLE IF EXISTS developer_links;
DROP TABLE IF EXISTS developers;
DROP TABLE IF EXISTS employers;
DROP TABLE IF EXISTS companies;
DROP TABLE IF EXISTS categories;
DROP TABLE IF EXISTS users;

CREATE TABLE users (
	user_id SERIAL PRIMARY KEY,
	user_name VARCHAR(30) NOT NULL,
	user_surname VARCHAR(50) NOT NULL,
	user_country VARCHAR(100) NOT NULL,
	user_email VARCHAR(100) NOT NULL UNIQUE,
	user_phone VARCHAR(18) NOT NULL
);

CREATE TABLE developers (
	dev_id INT PRIMARY KEY REFERENCES users(user_id) ON DELETE CASCADE,
	is_searching_job BOOLEAN NOT NULL DEFAULT TRUE
);

CREATE TABLE developer_links (
    link_id SERIAL PRIMARY KEY,
    dev_id INT NOT NULL REFERENCES developers(dev_id) ON DELETE CASCADE,
    link_url VARCHAR(255) NOT NULL
);

CREATE TABLE companies (
    company_id SERIAL PRIMARY KEY,
    company_name VARCHAR(150) NOT NULL UNIQUE
);

CREATE TABLE employers (
	employer_id INT PRIMARY KEY REFERENCES users(user_id) ON DELETE CASCADE,
	company_id INT REFERENCES companies(company_id) ON DELETE SET NULL,
	employer_position VARCHAR(100) NOT NULL
);

CREATE TABLE categories (
    category_id SERIAL PRIMARY KEY,
    category_name VARCHAR(50) NOT NULL UNIQUE
);

CREATE TABLE projects (
	project_id SERIAL PRIMARY KEY,
	dev_id INT NOT NULL REFERENCES developers(dev_id) ON DELETE CASCADE,
	category_id INT REFERENCES categories(category_id) ON DELETE SET NULL,
	project_name VARCHAR(150) NOT NULL,
	project_description TEXT NOT NULL,
	project_url VARCHAR(255) NOT NULL,
	project_active BOOLEAN NOT NULL DEFAULT TRUE
);
```
